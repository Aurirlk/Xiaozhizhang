# 新一代多模态 Agent 语音服务演进规划 (Phase 4 & Beyond)

基于目前已完成的 RESTful/WebSocket 纯净软件架构底座（含工厂模式、熔断器、成本控制、异步 CRM），项目已具备极高的扩展性。为向“多模态（Multimodal）”与“智能体（Agent）”演进，需要在现有架构上进行以下核心模块的迭代与功能补充。

## 一、 核心缺失功能补充 (Immediate Next Steps)

目前架构主要实现了线性的 `语音 -> 文本 -> 文本 -> 语音` 流程，距离真正的智能交互还有距离，需优先补充以下机制：

### 1. 意图打断机制 (Barge-in / Interruption)

- **当前痛点**：TTS 语音一旦开始合成下发，用户如果觉得 AI 说得太啰嗦，无法在中途打断，必须等它说完。

- **演进方案**：在 `ws_chat.py` 的全双工通信中加入**全时聆听（Full-duplex listening）**。当服务端正在下发 TTS 音频流时，持续接收客户端传来的音频帧；一旦服务端的 VAD（静音检测）模块识别到用户开始说话，立刻触发 `abort` 信号，截断当前的 TTS 发送，并清空当前播放队列，重新开始识别用户的打断指令。

### 2. 流式输入与端点检测 (VAD 实时切片)

- **当前痛点**：`/chat/audio` 接口目前接收的是一个完整的 `UploadFile`，这意味着用户必须“说完并松开按钮”系统才开始处理，延迟较高。

- **演进方案**：集成类似 `Silero VAD` 的算法，在 WebSocket 路由中实时接收碎片的音频流（如每 100ms 一个 chunk）。当 VAD 检测到用户停顿超过 800ms 时，自动进行断句（Sentence Segmentation），并将这一段发给 ASR，实现真正的“边听边想”。

### 3. 工具调用与插件系统 (Function Calling / Tool Use)

- **当前痛点**：LLM 只是“陪聊”，无法执行实质性任务（如查天气、播放指定歌曲、控制智能家居）。

- **演进方案**：这是迈向 Agent 的最关键一步。
  
  - 引入大模型原生的 `Function Calling` 能力。
  
  - 开发 `app/plugins/` 目录，封装诸如 `get_weather(city)`, `search_web(query)`, `control_device(device_id, action)` 等工具。
  
  - 交互流程变更为：ASR -> LLM (触发工具调用) -> 后端执行对应 Plugin -> 将结果返给 LLM -> LLM 生成最终回答 -> TTS。

## 二、 多模态 (Multimodal) 演进路线

单靠语音是不够的，引入视觉能力能让你的 Agent 具备真正的“感知”能力。

### 1. 视觉大模型接入 (VLLM Integration)

- **架构补充**：在 `ServiceFactory` 中增加 `create_vllm()`，接入如 `GPT-4o`、`Qwen-VL`、`GLM-4V` 等视觉语言模型。

- **接口改造**：升级 `/api/v1/ask` 接口，支持同时上传 `audio_file` 和 `image_file`。

- **业务场景**：
  
  - **看图说话**：“帮我看看这道菜叫什么？”（上传图片+语音）
  
  - **实时摄像头流分析**：客户端定时抽取视频帧（如每秒一帧）转为 Base64 发送给 WebSocket，Agent 结合当前画面主动发起语音对话（如：“你是不是在找钥匙？它就在你左手边的桌子上”）。

### 2. 情感分析与多模态 TTS 映射

- **架构补充**：
  
  - ASR 阶段除了提取文字，同时提取用户的**声学特征**（语速、音高、是否在哭泣/大笑）。
  
  - 视觉阶段提取用户的**面部微表情**。

- **TTS 联动**：将提取到的情绪标签（如 `emotion: sad`）连同文本一起丢给 MiMo TTS，让合成出来的语音带有对应的安抚情绪，实现真正的拟人化共情。

## 三、 深度记忆与个性化 (Deep Memory & Personalization)

你目前已经设计了优秀的 `AI CRM` 和 `对话历史管理`，但随着用户数据增加，直接传入历史记录会导致 Token 爆炸。

### 1. 向量数据库接入 (Vector RAG)

- **技术引入**：集成轻量级向量库（如 `ChromaDB`、`Milvus` 或 OceanBase 开源的 `PowerMem`）。

- **重构逻辑**：
  
  - 后台提取到的 CRM 用户画像和日常琐事（“我今天吃了一碗拉面”），通过 Embedding 模型向量化后存入数据库。
  
  - 当用户开启新对话时，系统先去向量库中检索**语义相关**的记忆（Semantic Search），然后作为 Context 注入给 LLM，取代暴力的全量历史拼接。

### 2. 长短期记忆融合 (Memory Manager)

- **短期记忆 (Short-term)**：使用 Redis 维护当前 session 最近 5-10 轮对话，保证连贯性。

- **长期记忆 (Long-term)**：使用向量库存储用户的核心画像（CRM）和历史事件。

- **工作流**：Agent 在回答前，不仅结合短期记忆，还会自主查询长期记忆（例如用户问“你还记得我最喜欢的电影吗？”，Agent 去 RAG 库中提取对应记录）。

## 四、 架构优化与运维迭代

为了应对后期的高并发与服务稳定性，系统底座需进一步巩固：

1. **消息队列解耦 (Celery + RabbitMQ/Redis)**
   
   - 将当前的 `asyncio.create_task` (用于 CRM 分析) 升级为真正的分布式任务队列。
   
   - 好处：即便并发请求激增，也不会拖垮主服务的 CPU，保证语音流的绝对低延迟。

2. **多租户与 Token 隔离**
   
   - 如果你未来打算把这个服务开放给其他人用，需建立严密的 `AppID / AppSecret` 鉴权机制。
   
   - 每个用户独立统计 Token 用量，独立熔断。

3. **前端/客户端生态扩展**
   
   - 除了 Gradio UI，可以考虑封装标准化的 WebSocket JS SDK，或者用 Flutter 快速套壳一个移动端 App，彻底取代原本的 ESP32 硬件载体。
